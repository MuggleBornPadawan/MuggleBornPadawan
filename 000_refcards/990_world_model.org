** clojure and webgpu
- the webgpu api specification (w3c): dictates the strict requirement for arraybuffer and typedarray usage for memory mapping
- stop trying to make clojurescript execute the math. the real power move is to use clojurescript purely as a macro language to dynamically generate wgsl code at runtime. you can write elegant, data-driven code that compiles into ultra-fast, static compute shaders. clojurescript becomes a meta-programming layer for the gpu, giving you functional ergonomics with bare-metal speed
- write all per-frame hot paths — chunk buffers, vertex data, uniform updates — in raw js interop using js/float32array, js/uint32array, and direct webgpu buffer mapping. let clojurescript own world state, event logic, and asset management outside the render loop. the gpu pipeline performs; clojurescript never enters the critical path
- the flaw: the most significant existential threat here is memory alignment and endianness. webgpu wgsl struct memory layouts are notoriously strict. if you are packing chunk buffers and uniform updates into a js/float32array, you must manually calculate byte offsets and padding (e.g., a vec3 in wgsl often requires the memory alignment of a vec4)
- the refinement: you must implement a robust, defensive serialization abstraction. you should write a macro or a pure js helper class that strictly manages the byte-stride and padding of your float32array so your clojurescript code never accidentally writes a float into a padding boundary
