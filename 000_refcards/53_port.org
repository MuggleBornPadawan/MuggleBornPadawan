** genai code limitations 
- q: given that ai is generating most of the code, is it even useful to learn new languages deeply?
  - a: You should learn at least one programming language really well - like, really, really well - and know the specifics of others
    - for example, if you are building something in go for the first time, you can get far by prompting your way through it. you can ship working code without knowing the basics
    - but yes, you can read the code, but can you truly understand it? let me double down...
    - go has goroutines and channels. rust has ownership and lifetimes. these are not syntax details; they are mental models
    - ai can generate code in a language you do not know, but it cannot generate intuition for a language you have never understood
    - when a deadlock happens, memory usage balloons, or a race condition surfaces, you need that mental model to reason about it. ai cannot hand you that reasoning
    - according to me, what ai changes is the ramp-up time - more specifically, syntax friction. that cost is now close to zero. the remaining work is the interesting part: building intuition for how the language thinks
    - so, you do not need to memorize every standard library method in every language. but you do need to deeply understand at least one language and then pick up the specifics of others as needed
    - that depth in one language gives you the mental model to reason about all the others
    - as engineers, our job is to solve problems, not necessarily to write code
- source: https://x.com/arpit_bhayani/status/2065063532232835494?s=20
*** solution/s   
  - a senior architect knows that ai-generated code written by someone without intuition is a high-interest technical debt engine. it creates "franken-code"—syntactically flawless scripts that fail catastrophically under load because the underlying concurrency or memory topology was completely misunderstood
  - status quo: ai accelerates "time to first commit" but the "time to production-grade verification" remains static or widens if the developer lacks domain intuition

*** core computer science invariants across languages
**** concurrency (time)
- the underlying truth across all the below five concurrency invariants exist because shared mutable state requires serialization of access that physical hardware does not provide automatically. every concurrency primitive — mutex, channel, actor, semaphore — is a mechanism for enforcing one or more of these invariants
  - mutual exclusion: at most one thread of execution may hold a lock on a shared resource at any given time. violation produces data races — two writers interleaving produces neither writer's intended state.
  - progress: if no thread holds a lock, some thread requesting it must eventually acquire it. violation produces livelock — threads are active but no work advances.
  - no starvation: every thread that requests a resource must eventually receive it. violation produces indefinite postponement — one thread monopolizes access while others wait forever.
  - happens-before: if operation a happens-before operation b, every thread that observes b must also observe a's effects. this is the invariant cpus and compilers violate via instruction reordering, and why memory barriers exist. violation produces visibility bugs — a write by thread 1 is not seen by thread 2 even after synchronization that looks correct.
  - deadlock freedom: no cycle can exist in the resource wait graph. if thread a waits for b's lock while b waits for a's lock, neither can proceed. the cycle is the violation.
**** memory (space)
- the underlying truth: memory invariants exist because hardware gives you a flat address space with no intrinsic notion of ownership, validity, or lifetime. everything below is enforced by the language runtime, the type system, the gc, or the programmer — never by the hardware itself.
  - validity: a pointer must refer to memory that has been allocated and not yet freed. violation: use-after-free, null dereference. the memory at that address now belongs to a different logical object or the os
  - single ownership (or controlled aliasing): at any point in time, either exactly one owner has write access, or multiple readers exist with no writer. both simultaneously violates the invariant. this is what rust's borrow checker enforces at compile time. languages without it push enforcement to runtime (gc) or abandon it (c)
  - bounds integrity: access to a buffer must stay within its allocated bounds. violation: buffer overflow. the most historically exploited class of memory bug because it corrupts adjacent memory — and in systems languages, that adjacent memory may be executable code or a return address
  - lifetime soundness: memory must not be accessed after its lifetime ends. a reference to stack memory that outlives its stack frame is a dangling reference — the address is valid in hardware but the logical object is gone
  - no aliased mutation: two pointers to the same memory location cannot both be used to mutate it concurrently or interleaved without synchronization. this bridges memory and concurrency — it's the invariant that makes data races a memory problem, not just a scheduling problem
**** type (meaning - logical interpretation)
- the underlying truth is that the types are propositions. a value inhabiting a type is a proof that the proposition holds. type checking is proof verification. unsoundness means the proof system can prove false propositions — which means you can derive anything, which means the type system provides no guarantees
  - type safety: a value of type t must never be used as if it were type s without an explicit, checked conversion. violation: type confusion bugs. in unsafe languages, casting a float's bit pattern and interpreting it as an integer is legal at the hardware level — the type invariant is what says it's wrong
  - soundness: if the type system says an expression has type t, evaluating that expression must produce a value of type t at runtime. a type system that allows proofs of type correctness that can fail at runtime is unsound. java's generic erasure and covariant arrays are classic soundness violations — both allow arraystoreexception at runtime for code that type-checks statically
  - progress and preservation (the two halves of type safety, formal definition): progress: a well-typed program either produces a value or takes a step — it does not get stuck. preservation: if an expression has type t and takes a step to expression e', then e' also has type t. together these guarantee well-typed programs cannot reach undefined states
  - exhaustiveness: a pattern match or case analysis over a sum type must handle all constructors. violation: a runtime match failure that the type system could have caught. haskell and rust enforce this. java switch historically did not
  - parametricity: a polymorphic function f : ∀a. a → a can only be the identity function. the type variable a is opaque — f has no operations available on a except to pass it through. this is a constraint, and that constraint is informative — it narrows the possible behaviors to one. this is the invariant that makes generic code more trustworthy than concrete code in well-typed languages

* foundational programming concepts
- hashing – converting data into a fixed-size value using a hash function for fast lookup or comparison
- memoization – caching results of expensive function calls
- caching – storing data temporarily to serve faster on repeated access
- serialization: converting an object or data structure into a byte stream or string so it can be stored or transmitted
- deserialization – reconstructing the original object or data structure from that byte stream or string
- replication: copying data across multiple systems to improve availability and fault tolerance
- bit manipulation – using bitwise operations for compact/fast logic
- sharding – dividing large datasets into smaller chunks for efficiency
- divide & conquer – break, conquer recursively, merge
- atomicity: ensuring an operation either fully completes or doesn’t happen at all, with no partial changes
- load balancing: distributing workloads across multiple servers to optimize performance and avoid overload
- concurrency: running multiple tasks simultaneously, overlapping their execution to maximize efficiency
- garbage collection: automatically reclaiming memory by detecting and freeing unused objects
- mutex: a lock ensuring only one thread accesses a resource at a time to prevent conflicts
- event loop: a mechanism that handles asynchronous tasks by queuing and processing events in a single-threaded environment
- indexing: creating a lookup structure to quickly find data in a database or array
- rate limiting: restricting the number of requests a system handles in a given time to prevent overload
- checksum & crc – a value computed from data to verify its integrity after transfer or storage
- memory pooling – pre-allocating memory to reduce allocation overhead
- deadlock: a situation where two or more processes block each other, waiting for resources that the others hold
- lazy loading: delaying the initialization or loading of resources until they’re actually needed
- event sourcing: storing the state of a system as a sequence of events rather than a single snapshot
- race condition: when the outcome of a program depends on the unpredictable timing of multiple threads accessing shared resources
- mapreduce: a framework for processing large datasets by splitting them into smaller tasks (map) andcombining results (reduce)
- dependency injection: passing dependencies (like services or objects) into a component rather than having it create them internally
- fuzzing: testing software by throwing random, unexpected inputs at it to find bugs or vulnerabilities
