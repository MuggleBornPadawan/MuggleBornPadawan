    - changes that affect multiple files or components
  - thinking mode is best for:
    - complex logic problems
    - debugging difficult issues
    - algorithmic challenges

** sample claude.md file
CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

UIGen is an AI-powered React component generator that uses Claude to help users build React components with live preview. The app features:

- Chat-based component generation using Vercel AI SDK
- Virtual file system for managing component files (not persisted to disk)
- User authentication and project persistence
- Live preview of generated components
- Code editor with syntax highlighting

## Tech Stack

- **Frontend**: Next.js 15 (App Router), React 19, TypeScript
- **Styling**: Tailwind CSS v4
- **Database**: Prisma with SQLite
- **AI**: Anthropic Claude via Vercel AI SDK (@ai-sdk/anthropic)
- **UI Components**: Radix UI primitives
- **Testing**: Vitest with React Testing Library
- **Build**: Turbopack in development

## High-Level Architecture

### Core Systems

1. **Virtual File System** (`src/lib/file-system.ts`)
   - In-memory file management for React components
   - Serializable state (sent to client, persisted in DB)
   - Files are never written to disk; all state is ephemeral in-memory

2. **AI Chat Pipeline** (`src/app/api/chat/route.ts`)
   - Streams responses from Claude using Vercel AI SDK
   - Uses Anthropic prompt caching for system messages (ephemeral cache)
   - Tools available to Claude: `str_replace_editor`, `file_manager`
   - Max steps set to 4 for mock provider (without API key), 40 with key

3. **State Management**
   - **ChatContext** (`src/lib/contexts/chat-context.tsx`): Manages chat messages and input
   - **FileSystemContext** (`src/lib/contexts/file-system-context.tsx`): Manages virtual file system
   - Both contexts coordinate to sync file changes from AI tool calls

4. **Authentication** (`src/lib/auth.ts`)
   - JWT-based sessions using jose library
   - HttpOnly secure cookies
   - 7-day session expiration
   - Server-only operations (marked with "server-only" imports)

5. **Database** (`prisma/schema.prisma`)
   - **User**: Email, password (bcrypt), timestamps
   - **Project**: Name, userId (optional), serialized messages, serialized file data

### Data Flow

1. User sends message via chat interface
2. Message + serialized file system sent to `/api/chat`
3. Claude receives message + system prompt (cached) + current files
4. Claude can call tools to modify files (str_replace_editor, file_manager)
5. Tool calls update virtual file system in memory
6. Stream response back to client (includes tool results)
7. FileSystemContext processes tool calls to update local file system state
8. If authenticated & projectId exists, save messages + file data to DB
9. Client updates preview with new file contents

## Important Patterns

### Import Aliases
- All imports use `@/` alias pointing to `src/`
- Configure in `tsconfig.json`
- Example: `import { Button } from "@/components/ui/button"`

### AI Prompts
- System prompt in `src/lib/prompts/generation.tsx`
- Directs Claude to:
  - Create App.jsx as root component export
  - Use Tailwind CSS (not inline styles)
  - Use `@/` import paths for project files
  - Operate on virtual filesystem (no traditional directories)

### Tools for Claude
- Located in `src/lib/tools/`
- `str_replace_editor`: Edit existing files (find & replace)
- `file_manager`: Create/delete files and directories

### Context Providers
- `ChatProvider` wraps main app, provides chat state
- `FileSystemProvider` wraps content, manages virtual file system
- Both communicate via `useFileSystem()` hook to handle tool calls

### Server Actions
- Located in `src/actions/`
- Handle data mutations (create project, get projects, etc.)
- Protected by authentication checks

## Key Implementation Details

### Virtual File System Serialization
- `VirtualFileSystem.serialize()` returns plain object for JSON
- `deserializeFromNodes()` reconstructs from object
- Sent to client and back to server without persistence

### Component Generation Flow
- Claude creates files at paths like `/App.jsx`, `/components/Button.jsx`
- Paths use leading `/`, no traditional directory structure
- App.jsx must export default React component
- All imports relative to root use `@/` alias

### Middleware
- `src/middleware.ts` protects routes requiring authentication
- Verifies JWT token and sets user context

### Tool Call Handling
- ChatContext calls `useFileSystem().handleToolCall()`
- FileSystemContext processes `str_replace_editor` and `file_manager` calls
- Updates in-memory file system immediately for preview

## Testing Notes

- Tests use `vitest` with `jsdom` environment
- React components tested with React Testing Library
- Test files colocated with source: `Component.test.tsx` in same directory
- Mock provider used in development (returns static components without API key)

## Notes for Future Work

- File system is ephemeral—always persists via Prisma when user is authenticated
- No validation on component code (trust Claude output, users can edit manually)
- Mock provider prevents API calls in development—useful for testing UI without costs
- Prompt caching on system message reduces latency and API costs
